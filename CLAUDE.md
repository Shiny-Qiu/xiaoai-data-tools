# CLAUDE.md

本文件为 Claude Code（claude.ai/code）在此仓库中工作时提供指引。

## 仓库概览

该仓库是一组服务于标注运营流程的独立 Python 数据处理脚本，不是带包结构的应用工程，也不是可复用的 Python 库。当前仓库没有统一入口、共享工具层、依赖清单或自动化测试体系；实际工作方式是按脚本逐个执行，对本地表格数据做离线处理。

整体处理模式比较稳定：
1. 从 `原始数据` 目录读取 `.csv`、`.xls`、`.xlsx` 文件
2. 使用 pandas 按既定业务规则进行整理、筛选、分组或分配
3. 将结果统一输出到 `处理结果` 目录，输出格式为 `.xlsx`

`main.py` 只是 PyCharm 默认示例脚本，不属于业务链路，可忽略。

## 架构与代码组织

### 执行模型

代码库采用“脚本直跑”模式，而不是“模块化导入 + 统一 CLI 入口”模式。每个脚本通常都独立维护以下内容：
- 顶层业务常量
- 输入输出目录约定
- 日志初始化
- pandas 数据处理逻辑
- Excel 写出逻辑

这意味着修改时不要先入为主地做抽象化或公共封装。多数变更应先判断：
- 是单个脚本内部的局部规则调整
- 还是会影响整个标注处理链路的数据约定

### 数据流视角的整体结构

虽然几个脚本在代码层面彼此独立，但在业务上形成了前后衔接的数据流：

1. `小爱session例行数据整理.py`
   - 对原始 session 数据做字段整理、排序和拆分
   - 目标是把上游导出的原始表格整理成更适合后续处理的标准化结果

2. `是否废除topic（保留最少轮次topic）.py`
   - 根据日期、query 有效性、Agent 类型和 topic 类型规则，决定哪些 `topic_id` 需要保留
   - 产出带 `是否废除` 字段的结果数据

3. `理想态人力分配.py`
   - 以“已保留的 topic”为输入
   - 按 topic 维度计算工作量并给标注员分配任务
   - 最终将分配结果回填到 `标注员1`

因此，这个仓库更适合被理解为“围绕标注数据的离线处理流水线”，而不是一个通用软件系统。

## 核心脚本说明

### `小爱session例行数据整理.py`

这是原始 session 数据整理脚本，主要负责标准化输入结构。

核心行为：
- 读取 Excel/CSV 文件
- 按 `target_columns` 指定的固定列顺序重排列
- 将 `timestamp` 尝试转换为时间后排序
- 按 `add_session_id`、`timestamp` 排序
- 按每 580 行拆分输出多个 Excel 文件

重要约束：
- 该脚本对输入表头高度敏感，默认要求 `target_columns` 中列出的字段都存在
- 输出文件名是在原目标文件名基础上追加序号生成
- 即使输入是 CSV，输出仍然是 Excel

这个脚本本质上是“结构整理 + 顺序整理 + 物理拆分”步骤，不承载复杂业务决策。

### `是否废除topic（保留最少轮次topic）.py`

这是 topic 保留/废除判定脚本，是业务规则最集中的一层。

核心行为：
- 遍历 `原始数据` 目录中的 `.csv`、`.xls`、`.xlsx` 文件逐个处理
- 将 `标注日期` 尽量标准化为 `YYYY-MM-DD` 字符串，并只针对 `TARGET_DATE`（当前脚本写死为 `2026-02-02`）执行保留逻辑
- 先在目标日期范围内按 `topic_id` 分组检查 `query有效性`
- 只有当某个 `topic_id` 在目标日期下对应的所有行，其 `query有效性` 都等于 `不参评query` 时，才把这个 `topic_id` 在该日期下整体排除出候选集合
- 只要某个 `topic_id` 在目标日期下并非“整组全是不参评query”，即使其中部分行不是 `有效query`，也仍然正常进入后续保留流程
- 对剩余候选数据，按 `RETAIN_RULES` 中定义的 `(真实细分Agent, topic类型) -> 保留数量` 逐组保留 topic
- 对 `跨垂域topic` 再额外执行一次单独保留，保留数量由 `CROSS_DOMAIN_RETAIN_COUNT` 控制，且会排除已经被前面规则保留的 topic
- 初始化整表 `是否废除 = '是'`，最终仅把“目标日期内且 topic_id 命中保留集合”的行改写为 `否`
- 输出文件统一写入 `处理结果` 目录，文件名为原文件名追加 `_处理结果.xlsx`

最重要的实现语义：
- 保留和剔除的最小单位都是 `topic_id`，不是单行
- “不参评 query 整组剔除”优先级高于配额保留；但剔除条件不是“出现一条非有效query”，而是“该 topic 在目标日期下整组所有行的 `query有效性` 都为 `不参评query`”
- 只要某个 topic 中存在任意一行不是 `不参评query`，该 topic 就仍会参与后续配额保留
- 候选 topic 的优先级由该 topic 在当前筛选结果中的行数决定，按行数升序保留，即优先保留轮次较少、体量较小的 topic
- `跨垂域topic` 不是参与 `RETAIN_RULES` 的普通 topic 类型，而是在常规保留后单独补充保留
- `是否废除` 的写回只作用于目标日期命中的行；非目标日期即使 `topic_id` 相同，也保持默认值 `是`

实现细节与约束：
- 脚本当前会在前置校验中检查 `真实细分Agent`、`topic_id`、`topic类型`、`标注日期`、`query有效性`；如果缺少其中任一列，会直接记录缺列错误并跳过该文件
- 日期标准化先尝试 `pd.to_datetime(...).dt.strftime('%Y-%m-%d')`；若失败，则退回为字符串后取空格前第一段，因此对混合格式日期有一定容错，但不是严格清洗
- 当某条规则下候选 topic 数量不足时，脚本只会记录 warning，不会中断，也不会补足其他来源的 topic
- 如果 `原始数据` 文件夹不存在，脚本会自动创建该目录并提示用户放入文件后重试
- 如果 `处理结果` 不存在，脚本会自动创建
- 输出格式固定为 Excel，即使输入是 CSV

修改这个脚本时，要特别注意：任何目标日期、`query有效性` 判定条件、Agent/topic 配额或跨垂域保留规则的变化，都会直接影响下游人力分配脚本的输入集合。

### `理想态人力分配.py`

这是人力分配脚本，核心目标是在可配置约束下，把保留下来的 topic 分配给标注员。

核心行为：
- 读取包含 `是否废除` 的数据文件
- 仅保留 `是否废除 == '否'` 的行
- 以 `topic_id` 聚合 topic
- 基于 `CATEGORY_TIME` 将 `规范类别` 换算为预计时长
- 基于 `ANNOTATORS_CONFIG` 判断 topic 是否可由某位标注员承接
- 采用贪心方式，尽量将每位标注员的累计时长推近 `TARGET_MINUTES_PER_ANNOTATOR`
- 将分配结果回填至 `标注员1`

最重要的实现语义：
- 分配单位始终是整个 topic，不允许拆 topic
- 只有当某个 topic 内所有 `规范类别` 都属于某位标注员允许范围时，才允许分配
- 分配逻辑不是全局最优求解，而是基于当前遍历顺序和约束的启发式分配

因此，如果后续发现“分配结果不均衡”，不要先假设是 bug，也可能是当前贪心策略与 topic 结构共同造成的结果。

## 关键数据约定

后续修改时，优先关注以下字段约定是否被破坏：

- `topic_id`：多个脚本中的核心分组键
- `是否废除`：topic 保留结果，也是人力分配脚本的直接输入条件
- `标注日期`：topic 保留逻辑中的目标日期筛选依据
- `query有效性`：用于判断目标日期下某个 `topic_id` 是否整组全部为 `不参评query`，若是则整组剔除
- `规范类别`：人力时长估算与可分配性判断的基础字段
- `标注员1`：人力分配的最终写回列

这些字段不是简单的展示列，而是跨脚本传递状态的关键业务字段。

## 字段依赖清单

### `小爱session例行数据整理.py`

该脚本依赖固定输入列集合 `target_columns`。其中关键字段包括：

- `add_session_id`：排序主键之一，用于把同一 session 聚拢
- `timestamp`：排序主键之一，用于保证 session 内时序
- 其余字段：按既定列顺序输出，默认都要求存在

输入要求：
- 输入文件必须至少包含 `target_columns` 中全部字段，否则在 `df = df[target_columns]` 时会失败

输出特征：
- 输出列顺序严格等于 `target_columns`
- 输出文件会被拆分为多个 `*_整理后_N.xlsx`

### `是否废除topic（保留最少轮次topic）.py`

该脚本直接依赖以下字段：

- `真实细分Agent`：用于匹配 `RETAIN_RULES`
- `topic_id`：topic 级别保留与剔除的核心键
- `topic类型`：用于区分 `单轮session`、`单轮topic`、`多轮topic`、`跨垂域topic`
- `标注日期`：用于筛选 `TARGET_DATE`
- `query有效性`：用于判断目标日期下某个 `topic_id` 是否整组全部为 `不参评query`；若是则整组剔除

脚本会新增/写回的字段：
- `是否废除`：默认写为 `是`，命中保留规则的 topic 写为 `否`

隐含依赖：
- 默认认为同一 `topic_id` 的多行记录应作为一个整体处理
- 保留逻辑只对目标日期生效

### `理想态人力分配.py`

该脚本直接依赖以下字段：

- `是否废除`：仅处理值为 `否` 的记录
- `topic_id`：topic 聚合键，也是分配单位
- `规范类别`：用于时长估算与人员能力匹配
- `标注员1`：最终回填分配结果

脚本内部配置依赖：
- `CATEGORY_TIME`：定义每种 `规范类别` 的单位时长
- `ANNOTATORS_CONFIG`：定义每位标注员可承接的类别范围
- `TARGET_MINUTES_PER_ANNOTATOR`：定义每位标注员的目标分钟数

隐含依赖：
- 如果同一 `topic_id` 内包含多个 `规范类别`，则这些类别必须全部被同一位标注员覆盖，才能完成分配

输入：
- 目录：`原始数据`
- 文件类型：`.csv` / `.xls` / `.xlsx`
- 最小关键字段示例：

```text
真实细分Agent | topic_id | topic类型 | 标注日期 | query有效性
```

说明：
- `T001` 可能因命中保留规则被保留

```text
T020     | 否       | 工具控制导航 |
T030     | 是       | 内容       |
```

输出：
- 目录：`处理结果`
- 文件示例：`某输入文件_人力分配结果.xlsx`
- 关键结果示例：

```text
topic_id | 是否废除 | 规范类别   | 标注员1
T001     | 否       | 通用问答   | 郑平
T001     | 否       | 通用问答   | 郑平
```

说明：
- 仅 `是否废除 == 否` 的 topic 会参与分配
- 同一 `topic_id` 下的多行会被作为一个整体分配给同一人
- 未被分配的 topic，其 `标注员1` 可能保持为空

## 重要实现说明

- 仓库默认采用本地目录工作流：输入目录为 `原始数据`，输出目录为 `处理结果`
- 主要业务规则以内嵌常量形式存在于脚本顶部，例如 `target_columns`、`RETAIN_RULES`、`CATEGORY_TIME`、`ANNOTATORS_CONFIG`
- 运行反馈主要依赖 `logging`，没有参数解析层，也没有统一命令行接口
- 当前脚本之间存在一定重复逻辑，这属于仓库现状，不要默认把它重构为共享抽象
- 如果修改字段名、过滤条件、topic 保留逻辑或人员分配规则，应同步评估是否会影响后续脚本对输入数据状态的假设
- 由于没有自动化测试，任何业务规则调整后，最有效的验证方式通常是用一份真实样例文件跑完整脚本并检查输出表结构与关键字段结果
